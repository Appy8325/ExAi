# Page Inventory

This document specifies every route in Concourse's Phase-1 route inventory as a buildable page: its primary components, the entities/endpoints it reads or writes, who may open it and under what entitlement, and which states from [13-application-layout.md](13-application-layout.md) §5 it implements. **This document owns the per-route page spec — the fourth and final layer of the routing stack** (route map → nav → shell → page). It does not own the route map itself ([11-information-architecture.md](11-information-architecture.md) §4, which this doc specifies exactly once each), navigation placement ([12-navigation-structure.md](12-navigation-structure.md)), shell/state-taxonomy definitions ([13-application-layout.md](13-application-layout.md)), the concrete component catalog ([15-component-inventory.md](15-component-inventory.md), which this doc originates names for), or column-level data shape ([16-database-schema.md](16-database-schema.md)).

Scope: **96 routes** — the 89 routes of [11-information-architecture.md](11-information-architecture.md) §4 (Auth, Account, Organizer Console org + event scope, Exhibitor Portal org + event scope, Attendee App, Platform Admin) plus the 7 marketing routes of [46-marketing-site.md](46-marketing-site.md) §2, each specified **exactly once**, organized into per-surface tables that mirror doc 11 §4's structure and numbering.

## 2. How to Read This Inventory

**Columns**, identical in every table below:

| Column | Contents |
|---|---|
| **Route** | Suffix after the surface/scope base path stated in the table heading, matching [11-information-architecture.md](11-information-architecture.md) §4's own suffix convention |
| **Components** | Primary content-region components (shell chrome — `Header`/`Sidebar`/`Footer`, `OrgSwitcher`, breadcrumbs — is never repeated per-page; it is constant per [13-application-layout.md](13-application-layout.md) §2–3). Names are canonical from first use here; [15-component-inventory.md](15-component-inventory.md) inventories every one |
| **Data & endpoints** | Entities from [00-foundation.md](00-foundation.md) §7 read/written, and the concrete route(s) from [18-api-architecture.md](18-api-architecture.md) §5 (or the doc that owns them, when §5 explicitly defers — auth flows to doc 19/20, analytics aggregates to doc 32, announcements/notifications mechanics to doc 33) |
| **Access** | Permission string(s) (`resource:action`, per foundation §8, drawn from the exact strings in [18-api-architecture.md](18-api-architecture.md) §5) plus any `entitlement:` key from [08-feature-matrix.md](08-feature-matrix.md) §3. `role: platform:admin` is used verbatim for Platform Admin per foundation §8 |
| **States** | The five states of [13-application-layout.md](13-application-layout.md) §5, abbreviated **L** (Loading), **E** (Empty), **Err** (Error), **Off** (Offline), **P** (Populated), one clause each, specific to this page |

**Default conventions** (stated once, applied silently unless a row overrides them, so per-row notes only spell out what's *distinctive*):

1. **Offline default is blocking.** Per [13-application-layout.md](13-application-layout.md) §5.4, any route not on JP-2's offline-tolerant list gets a blocking, explicit offline notice — never a spinner. The offline-tolerant exceptions are exactly: Attendee `/`, `/explore`, `/schedule`, `/badge`, `/map` (read-from-cache); Portal `/capture` and Console `/check-in` (write-queued). Every row below for those seven routes states its specific cache/queue behavior; every other row's Offline cell says simply **"blocking (default)"** unless a page-specific nuance exists.
2. **Empty is a collection concept.** Single-entity detail/settings/form pages have no zero-rows state; their Empty cell reads **"n/a — detail page"** or **"n/a — form page."** Where a *sub-region* of a detail page (e.g., a notes list) can be empty, that is noted as a scoped inline empty, not a page-level Empty.
3. **Locked (entitlement-gated) is a Populated variant**, per [13-application-layout.md](13-application-layout.md) §5.5 — never a 404. Rows whose entitlement can be absent state the Locked rendering inside the **P** cell.
4. **Error follows the scoped/full-page split** of [13-application-layout.md](13-application-layout.md) §5.3: a failed sub-widget gets an inline card; only primary-data failure gets a full-page error. Cells below name whichever applies.
5. **403/404 are routing-level outcomes**, resolved before a page mounts (doc 11 §3.3) — not repeated as a state on every row.
